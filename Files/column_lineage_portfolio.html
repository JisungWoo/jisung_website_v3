1 The Problem

Enterprise data warehouses grow views on top of views across dozens of schemas. Nobody can trace a single reporting column back to its actual source field.

Schema Layers

A single reporting view with ~290 columns sits atop a 15-hop dependency graph spanning 9 schema layers and multiple parallel source systems (SAP, Oracle, Microsiga). Columns flow through reporting → data mart → design → transform → staging → master → source schemas, with DML boundaries (stored procedures) breaking the trace midway.

Challenges

  • Snowflake's lineage API returns only 5 hops per call — deep traces require multi-window traversal
  • DML boundaries — stored procedures break the DDL chain, requiring ACCESS_HISTORY bridging
  • Parallel source systems — BRP, CIP, ERP, P11, Microsiga, Oracle variants have near-identical logic but point to different sources
  • Scale — 9,000+ views × hundreds of columns each = millions of potential edges to compute
  • Semantic gap — graph traversal finds the chain, but can't explain why a column is derived or what the SAP field name means

2 Architecture Overview

A hybrid engine: Python for deterministic graph traversal, LLM for semantic enrichment. All intermediate state lives in local SQLite — Snowflake is used for API reads and final publish only.

⚡

V3 Graph Engine (Python)

Builds the chain skeleton via BFS traversal, DDL parsing with sqlglot, and DML source stitching. Deterministic, fast, repeatable.

🧠

LLM Classification (Cortex)

Fills in meaning: derivation types, actual SQL expressions, SAP table / field mappings. Runs on Snowflake Cortex with domain-aware system prompts.

💾

SQLite Intermediate Store

WAL-mode SQLite holds all traversal state — edges, chains, DDL cache, LLM tasks. Zero Snowflake compute costs during computation.

📤

Snowflake Publisher

Batch-inserts curated lineage into 3 final Snowflake tables with deduplication, MERGE upsert, and run ID tracking.

4 Graph Traversal Engine

Multi-window BFS that overcomes the 5-hop API limit to trace 15+ hops deep.

Multi-Window Frontier BFS

Snowflake's GET_LINEAGE() returns at most 5 hops per call. To trace 15+ hops, the engine uses a windowed BFS — each window extends the frontier by another 5 hops:

# Windowed BFS — each call gets next 5 hops
frontier = [target_view]
visited  = set()
hop_window = 0

while frontier and hop_window * 5 < max_depth:
    next_frontier = []
    hop_window += 1

    for obj_fqn in frontier:
        # Each call returns next 5 upstream hops
        rows = GET_LINEAGE(obj_fqn, 'TABLE', 'UPSTREAM', 5)

        for row in rows:
            # Accumulate distance across windows
            distance = row.distance + (hop_window - 1) * 5
            store_edge(src, tgt, distance)

            # Only extend frontier for deep, non-terminal objects
            if distance >= 4 and not is_source_schema(obj):
                next_frontier.append(obj)

    frontier = next_frontier
Why multi-window?

GET_LINEAGE(TABLE) returns 5 hops per call. A reporting view that sits at the top of a 15-hop dependency chain requires 3+ windows to reach terminal source tables. The distance accumulator ensures globally consistent hop counts.

Column Graph: All-Paths BFS with DML Bridging

Once the object graph is built, the engine traces individual columns through two path types:

for root_column in target_columns:
    frontier = [(schema, object, column)]
    visited  = set()

    while frontier:
        (s_sch, s_obj, s_col) = frontier.pop()

        # Path A — DDL trace: source is a VIEW → parse its DDL
        if s_obj in view_names:
            for deeper in col_graph.get((s_sch, s_obj, s_col)):
                frontier.append(deeper)

        # Path B — DML bridge: source is a TABLE → check ACCESS_HISTORY
        else:
            for dml_source in dml_lookup[(s_obj, s_col)]:
                frontier.append(dml_source)
                # If DML source is a view, continue DDL trace
Why all-paths?

A column might have multiple source columns (COALESCE, CASE WHEN, UNION). Tracing all paths captures every possible origin — not just the shortest path. This gives analysts the full picture of where data could come from.

Terminal Detection

The traversal stops naturally at source system schemas (SRC_*). These are external data share tables with no further upstream DDL. The engine detects them via schema prefix and never attempts to fetch their lineage.

5 DDL Parser (sqlglot)

Abstract syntax tree analysis of CREATE VIEW statements — resolving CTEs, subqueries, and aliases to find physical source tables.

Parse → Resolve → Map

For each view in the object graph, the parser:

  1. Fetches DDL via GET_DDL('VIEW', fully_qualified_name)
  2. Parses with sqlglot.parse(ddl, read='snowflake') into an AST
  3. Extracts all CTEs into a dictionary: cte_real_sources['CTE_NAME'] = {(schema, table)}
  4. Maps CTE columns to real source: cte_col_map[('CTE_NAME', 'COLUMN')] = (table, field, schema)
  5. Processes the outermost SELECT, resolving each column through aliases and CTEs

CTE / Subquery Resolution

The most complex part of the parser. Given nested CTEs:

WITH stg AS (
    SELECT order_id, material_num
    FROM staging_schema.staging_table
),
enriched AS (
    SELECT s.order_id, d.material_name
    FROM stg s
    JOIN master_schema.dim_material d ON s.material_num = d.material_sk
)
SELECT e.order_id, e.material_name FROM enriched e

The parser recursively resolves:

Outer ColumnCTE ReferenceResolved Physical Source
e.order_idenriched → stg → staging_tablestaging_schema.staging_table.order_id
e.material_nameenriched → dim_materialmaster_schema.dim_material.material_name

Passthrough View Detection

Views prefixed with TV_ are passthrough wrappers — every column is a direct 1:1 mapping from the underlying physical table. The parser short-circuits these: all columns are classified as "direct" without deep AST analysis.

Column Graph Stitching

After all views are parsed, the engine builds a unified column graph and stitches it with DML bridge data from ACCESS_HISTORY. This creates a single traversable graph that spans both DDL-implicit paths (view definitions) and DML-implicit paths (stored procedures).

6 LLM Classification (Snowflake Cortex)

Domain-aware semantic classification — the graph engine finds the chain, the LLM explains the meaning.

System Prompt Engineering

The LLM receives a ~4KB domain-knowledge system prompt containing:

Source systems, table prefixes (TV_ = passthrough), SAP table catalog, system variant patterns (_BRP, _CIP, _ERP).

Cortex API Call Pattern

SELECT SNOWFLAKE.CORTEX.COMPLETE(
    'model-name',
    [
        {'role': 'system', 'content': system_prompt},  -- ~4KB domain knowledge
        {'role': 'user',   'content': 'Schema: ... View: ... DDL:\n' || ddl_text}
    ],
    {'temperature': 0.1, 'max_tokens': 16384}
) INTO v_response;

Token Management & Cost Control

  • Compact JSON output — system prompt mandates minified JSON (60% token savings vs. pretty-printed)
  • DDL cache — each view's DDL fetched once, cached with MD5 hash; skip reclassification if hash unchanged
  • Incremental processing — work queue skips already-classified views unless RECLASSIFY=TRUE
  • Per-call logging — every Cortex call logged with tokens_in, tokens_out, duration_ms, status

Truncation Recovery

Large views with 200+ columns can exceed the output token limit. Instead of losing all results, the engine recovers partial output:

-- If JSON parse fails and output starts with '['
IF (v_parsed IS NULL AND LEFT(v_message_text, 1) = '[') THEN
    -- Find the last complete JSON object boundary: '},{'
    v_last_brace := REGEXP_INSTR(text, '\}\s*,\s*\{', ...);

    -- Truncate at the last complete object, close the array
    v_message_text := LEFT(text, close_pos) || ']';
    v_parsed := TRY_PARSE_JSON(v_message_text);  -- Retry
END IF;
Why recover instead of retry?

Retrying the same view with the same DDL will hit the same token limit. Recovery salvages the first N complete column classifications (e.g., 180 of 250 columns) rather than getting 0. The remaining columns can be classified in a follow-up targeted run or with a larger max_tokens setting.

WAL Mode Benefits

conn.execute("PRAGMA journal_mode=WAL")  -- Write-Ahead Logging
conn.execute("PRAGMA foreign_keys=ON")  -- Referential integrity
  • Concurrent reads + writes — readers never block writers, critical for multi-phase pipeline
  • Crash safety — WAL file preserves committed data even on interrupted runs
  • Incremental re-runs — clear_for_target() deletes one view's data without corrupting others
  • Transaction atomicity — all-or-nothing inserts per phase
Why SQLite instead of Snowflake for intermediate state?

Graph traversal involves thousands of small reads/writes per second — perfect for local SQLite but expensive on a cloud warehouse. The engine runs heavy computation locally at zero cloud cost, only touching Snowflake for API reads (Phase 1) and final publish (Phase 3).

8 Key Engineering Decisions

Trade-offs that shaped the architecture.

🔄 Hybrid V3 + LLM

Decision: Graph engine builds the chain skeleton; LLM fills in semantic meaning.
Why: Pure LLM can't traverse 15-hop graphs reliably. Pure graph can't explain derivation types. The hybrid gives deterministic structure + rich semantics.

📊 All-Paths vs. Shortest-Path

Decision: Trace all possible source paths, not just the shortest.
Why: A COALESCE(A, B, C) column has 3 valid sources. Analysts need to see all possibilities, not just the primary one.

🔌 DML Bridge Pattern

Decision: Don't trace through tables during graph walk. Use ACCESS_HISTORY in a separate batch phase.
Why: One batch query across all tables is far faster than per-table ACCESS_HISTORY calls. Separating concerns also simplifies debugging.

💨 DDL Parse as Default (Not Per-Column API)

Decision: Parse DDL locally with sqlglot by default; per-column API only for verification.
Why: 1 GET_DDL + local parse vs 290+ GET_LINEAGE COLUMN calls. 15x faster for the same result.

🗃️ Persistent DDL Cache

Decision: Cache DDL text + MD5 hash. Skip reclassification when hash unchanged.
Why: Most views don't change between runs. Why pay for LLM tokens on identical DDL?

🧩 Decoupled LLM Pipeline

Decision: LLM enrichment is optional phases (2.5/2.6/2.7), not inline with graph traversal.
Why: Lineage can run without waiting for LLM. LLM results are cached and reusable. Different models can be compared without re-running the graph engine.

🔢 Run ID & Idempotence

Decision: Every run gets a UUID. Target-level clear before re-insert.
Why: Enables comparing multiple runs of the same view, auditing changes over time, and safe re-runs without data corruption.

🛡️ Truncation Recovery

Decision: Salvage partial LLM output instead of failing entirely.
Why: Large views (200+ columns) may exceed output token limits. Getting 180 of 250 columns classified is infinitely better than getting 0.
